# 热门视频榜单 Agent — 产品架构设计 v2

> 版本：v2（基于可行性评估优化）
> 日期：2026-03-14

---

## 一、产品定位

自动爬取视频平台（MVP 以抖音为主）在特定赛道（MVP 以知识传播/卖课为主）内的热门视频榜单，输出：

- **日榜 Top10**：前一天播放量增量前十的视频
- **周榜 Top10**：上一整周播放量增量前十的视频
- **月榜 Top20**：上一个月播放量增量前二十的视频

---

## 二、核心问题修正：视频发现层

> v1 的致命缺口：没有解释"如何找到知识卖课赛道的视频"。
> 增量计算的前提是已在追踪某个视频，但候选视频池从哪里来？

### 视频发现策略（三路入口）

```
┌─────────────────┬─────────────────────────────────────────┬──────────────────┐
│    发现入口      │                  方式                   │      覆盖度       │
├─────────────────┼─────────────────────────────────────────┼──────────────────┤
│ 头部账号追踪    │ 维护知识博主账号白名单，监控新发视频     │ 高（精度高）      │
├─────────────────┼─────────────────────────────────────────┼──────────────────┤
│ 关键词搜索      │ 搜索"知识付费/副业/学习/干货"等词结果   │ 中               │
├─────────────────┼─────────────────────────────────────────┼──────────────────┤
│ 话题/挑战赛页面 │ 爬 #干货 #副业 #知识 等话题下的视频列表  │ 中               │
└─────────────────┴─────────────────────────────────────────┴──────────────────┘
```

三路结果统一经过去重（布隆过滤器）→ 赛道分类过滤器 → 进入追踪快照池。

## 五、核心数据模型

```python
# 视频基本信息
Video:
    platform: str          # "douyin" | "bilibili"
    video_id: str          # 平台原始 ID
    track: str             # "knowledge_course" | ...
    title: str
    author_id: str
    author_name: str
    track_confidence: float
    classify_stage: str    # "rule" | "tag" | "llm"
    discovery_source: str  # "account" | "keyword" | "topic"
    is_tracking: bool
    published_at: datetime

# 视频快照（时序表，增量计算的数据基础）
VideoSnapshot:
    video_id: int          # FK → videos.id
    play_count: int
    like_count: int
    comment_count: int
    share_count: int
    collect_count: int
    snapshot_at: datetime  # 精确到整点小时，消除采集漂移

# 榜单
Ranking:
    period_type: str       # "daily" | "weekly" | "monthly"
    platform: str
    track: str
    period_start: date
    period_end: date
    top_n: int             # 10 或 20

RankingItem:
    ranking_id: int
    rank: int
    play_increment: int    # 周期内增量播放数（排序依据）
    play_count_end: int    # 期末绝对播放数
    video_title: str       # 冗余存储，防止视频删除后数据丢失
    author_name: str
```

## 六、业务语义定义（已锁定）

| 榜单 | 排序依据 | 计算方式 | 快照时间点 |
|------|---------|---------|-----------|
| 日榜 Top10 | 24h 播放增量 | T(今日0点) − T(昨日0点) | 每日 00:05 触发 |
| 周榜 Top10 | 7日播放增量 | T(本周日23:59) − T(上周日23:59) | 周一 00:10 触发 |
| 月榜 Top20 | 当月播放增量 | T(月末23:59) − T(月初00:00) | 每月1日 00:10 触发 |

> **月榜排序依据：当月播放增量**（而非月末绝对播放量），与日/周榜口径一致。

| monthly_ranking | `30 0 1 * *` | RankingGenerator.generate(MONTHLY)（每月1日00:30）|

## 八、赛道分类器设计（三阶流水线）

```
视频输入
   │
   ▼
Stage1: 关键词规则匹配
   ├─ 命中且置信度 ≥ 0.8 → 直接返回结果（最快，O(1)）
   └─ 未命中 ↓
   ▼
Stage2: 账号标签匹配
   ├─ 账号在白名单且已标注赛道 → 置信度 1.0，直接返回
   └─ 未命中 ↓
   ▼
Stage3: Gemini API 分类（兜底）
   └─ 输出 {label, confidence, reason}
```

- Stage1/2 覆盖约 80% 的视频，成本接近零
- Stage3（Gemini）仅处理剩余 20%，控制 API 成本

## 九、存储规模估算

```
追踪视频池估算（知识卖课赛道）：
  头部账号追踪  ~500账号 × 50视频  =  25,000 条
  关键词搜索补充                    +   5,000 条
  话题页发现补充                    +   5,000 条
  ─────────────────────────────────────────────
  候选池总量                       ~  35,000 条

快照存储估算：
  35,000 条 × 2次/天 × 365天 = 2,555万行/年
  单行约 200 bytes → ≈ 5GB/年

结论：PostgreSQL 按月分区索引完全够用
```

## 十一、扩展性设计原则

1. **平台扩展**：新增平台只需实现 `PlatformDataSource` 接口，注册到 `factory.py`，上层零改动
2. **赛道扩展**：赛道配置化（`config/tracks.yaml`），支持关键词热更新，无需重启
3. **榜单维度扩展**：`PeriodType` 枚举化，新增周期只需添加枚举值和 Cron Job
4. **通知渠道扩展**：`NotificationChannel` 接口 + Observer 模式，新增渠道无侵入
